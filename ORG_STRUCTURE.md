# StartX · 组织与部门系统（规模化经营）

> 解决"公司超过 ~100 人后，桌面塞爆、微操爆炸"的核心可扩展性问题。
> 思路：**用部门卡把人折叠**，让桌面上的卡数恒定（~6–10 张），与实际人数解耦。
> 这也是本作从"Stacklands 式微操"过渡到"宏观经营"的核心机制阀门。
> 关联：[RESEARCH_TREE.md](RESEARCH_TREE.md)、[RECIPE_BOOK.md](RECIPE_BOOK.md)、[BOOSTER_PACKS.md](BOOSTER_PACKS.md)。

---

## 1. 为什么需要部门：可扩展性问题

| 人数 | 若用个人卡 | 体验 |
|------|-----------|------|
| 1–10 | 10 张卡 | ✅ 爽，Stacklands 手感 |
| 10–30 | 30 张卡 | ⚠️ 桌面拥挤，开始烦 |
| 30–100 | 100 张卡 | ❌ 桌面爆炸、找不到人、无法微操 |
| 100+ | 不可玩 | ❌❌ |

**结论**：卡数不能随人数线性增长。需要分层折叠，使**可见卡数 ≈ 组织单元数（部门）**，而非人数。

---

## 2. 四层组织抽象（分层折叠）

```
个人卡(1) ──聚合──▶ 团队/班组(2–7) ──组建──▶ 部门(8–30) ──合并──▶ 事业部(多部门) ──▶ 公司
   早期手把手          中期带组长           1 张卡·自动产线       后期 1 张卡         整盘
```

| 层级 | 人数 | 桌面形态 | 操作粒度 |
|------|:---:|----------|----------|
| 个人 Employee | 1 | 单卡 | 手动拖到节点/配方 |
| 团队 Team | 2–7 | 带 Lead 的小栈 | 半自动，拖整栈 |
| **部门 Department** | 8–30 | **1 张部门卡** | 绑定流水线，自动运行 |
| 事业部 Division | 多部门 | 1 张事业部卡 | 设目标/预算，部门自治 |
| 公司 Company | 全部 | 棋盘本身 | 战略/季度决策 |

**关键不变量**：100 人 ≈ 桌面上 6–10 张部门/事业部卡。空间占用恒定。

## 3. 组建部门（折叠触发）

- **触发条件**：同职能（workTag 相同）员工数 ≥ `formThreshold`（建议 6）时，弹出"组建部门"提示（可视为一个组织 Idea）。
- **组建配方**：`≥4 同职能员工 + 1 经理 + 1 设施/楼层` → 折叠成 1 张**部门卡**。
- 部门卡内部存 `members[]`、类型、经理、等级；桌面只显示聚合信息（人数 / 产能 / 状态）。
- **可逆**：拆分部门 → 释放回个人卡（重组，见 §7）。

### 部门类型（对应职能/workTag）
研发部 · 产品部 · 销售部 · 市场部 · 交付/运营部 · 客户成功部 · 财务部 · 法务部 · 人力部 · 战略投资部

## 4. 部门 = 自动生产引擎

部门卡绑定**一条流水线**（一个保存的配方链），自动运行：

- **输入**：从相邻"吸附输入区"自动取料（如销售部吃"线索"）。
- **输出**：产出到"输出栈"或现金。
- **吞吐公式**：
```
部门产出/秒 = baseRate × 有效人力 × 经理倍率 × 设施倍率 × 装备倍率
有效人力   = 人数 / (1 + 协调开销)
协调开销   = max(0, 人数 − 管理幅度 × 经理数) × k     # k≈0.08
```
- 即：人越多产能越高，但**边际递减**；靠经理/制度/设施抵消开销。

---

## 5. 管理幅度与科层（防无脑堆人）

- **管理幅度 Span**：1 名经理可有效管理约 **7** 人（`span=7`）。
- 超过 `span × 经理数` 的人进入"失管"，贡献协调开销（效率掉）。
- 解决：**加中层经理** → 自动形成子团队/科层 → 部门升级（T1 小组 → T2 部门 → T3 大部门/中心）。
- 这把"组织设计"变成可玩选择：扁平(快但脆) vs 科层(稳但贵)。

| 部门等级 | 人数上限 | 需经理数 | 特性 |
|:---:|:---:|:---:|------|
| T1 小组 | 7 | 1 | 灵活、开销低 |
| T2 部门 | 20 | 1 + 2 中层 | 标准产线 |
| T3 中心/大部门 | 40 | 1 总监 + 多中层 | 高产能、高固定成本 |

---

## 6. 部门 = 风险防御单元

- 可指派部门**自动抵御某类风险**：
  - 法务部 → 监管问询 / 合同纠纷 / 数据泄露(合规面)
  - 客户成功部 → 客户投诉 / 续约流失
  - 财务部 → 现金流危机 / 审计调整
  - 市场部 → 舆情危机 / 价格战(品牌面)
- 风险进入棋盘时，对应部门若已设防则自动处置（消耗部分产能/工时），无需手动拖人。
- 与战斗系统接口：部门聚合 UnitStats 后整体应战。

---

## 7. 重组 Reorg（战略决策）

- **操作**：拆分 / 合并部门、调岗、增设中层、撤销事业部。
- **成本**：一次性**士气下降** + 短期产能波动（真实的"重组阵痛"）。
- 做成**季度决策卡**之一：增长期合并提效、危机期拆分止血。

## 8. 与各阶段结合

| 阶段 | 组织形态 | 典型动作 |
|------|----------|----------|
| 起步/产品 | 全个人卡（≤10 人）| 手把手，享受 Stacklands 手感 |
| 经营 | 出现首批团队/部门（10–30）| 组建研发部、销售部，绑定产线 |
| 资本 | 部门化为主（30–60）| 加中层、建财务/法务部防风险 |
| 扩张 | 多部门 + 首个事业部（60–120）| 按业务线分事业部、自动化产线 |
| 终局 | 事业部 + C-suite（120+）| 集团化、并购整合、季度战略 |

> 折叠时点与卡包/研发解锁挂钩：如研发节点 **F1 自动化蓝图** 解锁后，部门可执行"一键工作流"。

## 9. 数据模型（实现参考）

```ts
type OrgUnit = {
  id: string;
  kind: 'employee'|'team'|'department'|'division';
  name: string;             // '研发部'
  func: WorkTag;            // 'research'|'sales'|...
  members: string[];       // 下级 OrgUnit / 员工 instanceId
  manager?: string;        // 经理卡 id
  headcount: number;       // 递归汇总
  level: 1|2|3;
  span: number;            // 管理幅度，默认 7
  pipelineId?: string;     // 绑定的流水线（配方链）
  defends?: RiskClass[];   // 自动防御的风险类别
  facility?: string;       // 所在设施/楼层
  equipment?: string[];
};

// 有效产能
function throughput(u: OrgUnit): number {
  const managers = countManagers(u);
  const overhead = Math.max(0, u.headcount - u.span * managers) * 0.08;
  const effective = u.headcount / (1 + overhead);
  return BASE_RATE * effective * mgrMult(u) * facilityMult(u) * equipMult(u);
}
```

**月度结算**：按 OrgUnit 递归聚合工资与管理成本，HUD 只展示部门级数字。

## 10. 与当前 MVP 切片的衔接（后续实现步骤）

当前切片是个人卡 + 手动堆叠。扩展为组织系统：

1. **`OrgUnit` 数据结构 + 折叠/展开**：部门卡是一种特殊 Card，点击可"展开"抽屉查看/调配个人卡，默认折叠。
2. **组建/拆分配方**：达到 `formThreshold` 弹提示，执行折叠。
3. **部门自动产线**：复用现有 `RecipeResolver`，但驱动方为部门吞吐公式而非单卡。
4. **吸附输入/输出区**：部门旁的磁吸格（复用 GDD 的 MagnetRule / CRM 自动归堆思路）。
5. **HUD 组织面板**：组织图（树）视图，显示各部门人数/产能/防御。
6. **重组决策**：季度决策卡接入。

数据建议落在 `game/data/org.json`（部门模板、阈值、幅度、开销系数）。
